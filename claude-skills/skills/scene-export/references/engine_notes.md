# Engine notes and honest limitations

## Why OBJ and glTF needed no axis conversion, but a USD exporter would

This project's `ScenePlan` uses meters, Y-up, right-handed â€” the same
convention as glTF's own spec default and Unity/Unreal's common default.
That's why `export_obj.py` and `export_gltf.py` write positions directly
with no rotation/conversion step.

**USD (and therefore Omniverse) defaults to Z-up.** A USD exporter was
considered for this skill but NOT implemented, specifically because
getting this conversion right (and documenting exactly when it's needed
vs. already handled by the importing tool) is real, easy-to-get-wrong
work that deserves its own tested implementation rather than a rushed
addition. If/when a USD exporter is built, the axis conversion (swap
Y and Z, negate one axis to preserve handedness) needs the same level of
verification this skill's other exporters received â€” not just "add a
rotation and hope."

## Why primitive tessellation is duplicated between export_obj.py and reused, not export_gltf.py

`export_gltf.py` imports `_PRIMITIVE_MESH`, `_transform`, and
`_rotate_point` directly from `export_obj.py` rather than reimplementing
mesh generation â€” this was a deliberate choice to avoid maintaining two
separate (and potentially divergent) geometry implementations. If you
modify sphere/box/cylinder tessellation, both exporters pick up the
change automatically. `export_threejs.py` and `export_godot_gdscript.py`
do NOT reuse this code, because both target engines have their own
built-in primitive geometry (THREE.SphereGeometry, CSGSphere3D) â€” baking
custom vertex data into those would be pointless when the engine already
does it natively and typically better (smoother spheres, proper UVs).

## Capsule and cone are approximations, not real capsule/cone geometry, in the OBJ/glTF exporters

`_PRIMITIVE_MESH` maps `"capsule"` to the same icosphere function as
`"sphere"`, and `"cone"` to the same function as `"cylinder"` â€” a real
capsule (cylinder with hemispherical caps) and a real cone (tapering to
a point) are NOT implemented. This is disclosed in `export_obj.py`'s
`_PRIMITIVE_MESH` dict comments and repeated here because it's easy to
miss: if a qFoldIT visualization actually needs a true capsule or cone
shape (not just "a shape roughly capsule/cone in spirit"), this
exporter will silently give you a sphere or cylinder instead â€” check the
output geometry, don't assume the primitive name guarantees the exact
shape.

**In `export_threejs.py` and `export_godot_gdscript.py`, capsule and
cone ARE real** (`THREE.CapsuleGeometry`, and Godot's `CSGCylinder3D`
has a native `cone` boolean property) â€” this inconsistency between
exporters is itself worth knowing about, not just each individual gap.

## What "verified" means differently across the four exporters, and why

This project could not install `pygltflib`, run a real Godot instance,
or reach the internet to load Three.js inside this sandbox (no network
access). So "verified" means something different per exporter, and
SKILL.md's table states this explicitly rather than using one blanket
"tested" label for all four:

- OBJ: hand-written structural + geometric checks, no external OBJ
  parser available to cross-check against â€” but OBJ is simple enough
  that a hand-written check covering index-range validity and decoded
  vertex position correctness is a meaningfully strong test.
- glTF: hand-written spec-conformance checks against the actual glTF
  2.0 specification (buffer/bufferView/accessor byte-range and
  cross-reference validity), since no `pygltflib`/`jsonschema` was
  available to validate against an existing implementation. This is
  still not the same guarantee as running the file through the official
  glTF-Validator or loading it in an actual glTF viewer â€” recommend
  doing that yourself for anything high-stakes.
- Three.js: the ONE exporter whose actual visual output was confirmed
  by a real render, in a real browser (via this environment's
  Visualizer widget tool, which executes in the user's browser, not
  this sandbox) â€” during this skill's development, a methane molecule
  scene generated with the same logic as `export_threejs.py` was
  rendered and visually confirmed to show the correct tetrahedral
  geometry.
- Godot GDScript: the weakest guarantee of the four â€” brace-balance and
  known-class-name checks only, no real execution. Say this plainly
  whenever this exporter is used.

## Known gaps (not implemented, and not silently worked around)

- No animation of any kind.
- No texture/UV mapping â€” solid colors only.
- No skeletal rigs / bones.
- No physics colliders (though CSG shapes in Godot and most engine
  imports of OBJ/glTF can have collision auto-generated by the receiving
  engine â€” that's the receiving engine's job, not this exporter's).
- No USD/Omniverse exporter (see axis-conversion note above for why).
- No direct PDB-to-scene-plan converter â€” `build_molecule_scene` takes
  an already-parsed atom/bond list, not a raw PDB file. If a real PDB
  structure exists (e.g. from `qfoldit-bionemo-agent-toolkit`'s
  AlphaFold2 wrapper), parsing it into the `(element, x, y, z)` /
  `(atom_a, atom_b)` lists this function needs is a separate step not
  implemented here.
